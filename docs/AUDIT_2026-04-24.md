Worth fixing — cheap to add and the tables get written:

### 5b. Unused indexes (13 × INFO)

`idx_activities_location_mode`, `idx_activities_location_lat_lng`,

### 5c. Multiple permissive policies (11 × WARN)

`connect_codes`, `fcm_tokens`, `push_subscriptions` each have two
overlapping SELECT policies. Postgres evaluates both on every read.
Worth consolidating into a single policy with OR'd conditions. Pattern:

```sql
DROP POLICY "Anyone can read active codes" ON connect_codes;
DROP POLICY "Users can manage own codes" ON connect_codes;
CREATE POLICY connect_codes_read ON connect_codes FOR SELECT
  USING (status = 'active' OR user_id = (SELECT auth.uid()));
```

### 5d. Auth DB connections (INFO)

Auth server uses a fixed 10-connection allocation. Fine for current
scale; switch to percentage-based allocation before a bigger instance.

## 7. Row-level security (RLS) posture

All 19 `public.*` tables have RLS enabled. Summary:

- **Owner-read/write tables** (`user_contacts`, `saved_searches`, `notifications`, `fcm_tokens`, `push_subscriptions`, `reports`, `reviews`): `USING/WITH CHECK (auth.uid() = <owner_col>)`. All use `(SELECT auth.uid())` wrapping to hit the init-plan optimizer.
- **Public-read tables** (`profiles`, `follows`, `activities`): SELECT `USING (true)` for authenticated. Expected for a social product.
- **Visibility-aware** (`posts`): SELECT policy checks `visibility` enum against follow graph / ally status. Handles blocks via `user_contacts.status='blocked'`.
- **Anonymous-insert** (`connect_messages`, `connect_scans`): anon INSERT with `WITH CHECK (true)` — intentional for Contact Codes.

## 9. Known issues / follow-ups

- **Seed users are marked verified.** All 50 have `verified_email/phone/id/selfie = true`. For production realism, strip verification:
  ```sql
  UPDATE profiles SET verified_email=false, verified_phone=false, verified_id=false, verified_selfie=false
  WHERE email LIKE '%@seed.buddyally.local';
  ```
- **Nuke seed if/when ready:**
  ```sql
  DELETE FROM auth.users WHERE email LIKE '%@seed.buddyally.local';
  ```
  Cascades to profiles, posts, activities, etc.
- **Seed first names are lowercase** — `abdul`, `yinka`, etc. Cosmetic. Fix if keeping seed:
  ```sql
  UPDATE profiles SET first_name = initcap(first_name) WHERE email LIKE '%@seed.buddyally.local';
  ```
- **Saved-search notify fan-out** only fires on client-created activities, not on direct SQL inserts. If you want retroactive pings, call `/api/saved-search-notify` per matching activity.
- **Stray files** at repo root should be added to `.gitignore` or removed (see §3).
